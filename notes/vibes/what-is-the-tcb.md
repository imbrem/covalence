# What exactly is the TCB?

**Status:** human-readable orientation, derived from the machine artifacts —
the golden manifests (`docs/deps/{core,eval,builtins}-manifest.txt`) and the
audit (`docs/deps/tcb-audit.json`, generated by `scripts/tcb-audit.mjs`).
Numbers are a **snapshot (2026-07-11)**; the live source of truth is those files
(regenerate: `bun run deps`; print: `bun run tcb`). If a number here disagrees
with the JSON, the JSON wins.

**Companions:** [`kernel-design.md`](./kernel-design.md) (read first before
touching the TCB), [`closed-world-kernel.md`](./closed-world-kernel.md) (the base
design), [`base-abstraction.md`](./base-abstraction.md) (the facade + the future
data-shaped TCB), [`soundness-audit.md`](./soundness-audit.md) (the deeper audit).

## The TCB in one paragraph

The **Trusted Computing Base** is the code whose correctness you must believe for
a Covalence theorem to mean what it says. Everything else — front ends, drivers,
tactics, importers, the server, the CLI — is *untrusted*: it can only produce a
`Thm` by going through the TCB's audited mint sites, so a bug there yields a
*failure to prove*, never a *false theorem*. The TCB is deliberately small,
`unsafe`-free, and enumerable: it is the base equality kernel plus the HOL rules
built on it, and it grows *only* by the computation tier (`eval`) when you opt
into arithmetic/literals.

## The trusted crates (concentric configs)

The audit measures the TCB as **nested configurations** — you only pay for the
tier you use. Snapshot (tests excluded; lower = more auditable):

| config | crates | files | src-lines | `unsafe` | mint sites | admitted rules |
|---|---|---|---|---|---|---|
| **base** | `covalence-pure-trusted` (`crates/kernel/base/trusted`) | 14 | 1496 | 0 | 24 | — (equality calculus) |
| **base+HOL** | + `covalence-core` (`crates/kernel/hol/core`) | 56 | 6951 | 0 | 24 | 25 (`core`) |
| **base+HOL+eval** | + `covalence-hol-eval` (`crates/kernel/hol/eval`) + `covalence-pure-eval` (`crates/kernel/base/eval`) | 90 | 13281 | 0 | 24 | 25 + 17 + 387 |

Key reads:

- **Mint sites stay at 24 across every config.** The number of places a `Thm` is
  born (`Thm::new`) does *not* grow when you add HOL or eval — HOL/eval rules are
  *admitted rules routed through the base's `apply`/`canon`*, not new mint sites.
  This is the central soundness lever: **one mint chokepoint, audited once.** The
  base's own goal is *fewer* mint sites over time (the relations refactor wants
  ideally just `execute` + instantiation + transport;
  [`base-abstraction.md`](./base-abstraction.md) §7).
- **Zero `unsafe`** in every trusted config — a hard, CI-tracked invariant.
- The audit also tracks a **`base+HOL (target)`** config (files 36 / lines 5142 /
  `defs::` coupling 17 vs `base+HOL`'s 29) — the shrink goal as the literal-leaf
  `defs/` residue dies (root `SKELETONS.md` #6, the leaf-elimination thread).

## The 24 mint sites (where trust actually enters)

Every `Thm::new` in the base, by file (from `tcb-audit.json → mintSiteLocations`):

- **`trusted/src/eqn.rs` (12)** — the equality calculus: `refl`, `sym`, `trans`,
  `eq_mp` (Leibniz), `cong_app`, `cong_pair`, `of_ptr_eq`, `lift`, `of_eq`,
  `apply` (the general admitted-rule gate), `canon` (op evaluation).
- **`trusted/src/prop.rs` (5)** — the pure bool theory: `and_intro`, `or_inl`/
  `or_inr`, and/or elim, `mp`.
- **`trusted/src/rel.rs` (6)** — the positive relation calculus: `execute`
  (run untrusted `f`, mint `⊢ (a,b) ∈ Rel(f)`), `transpose`, `compose`, `meet`,
  `join` (×2). **Positive-only** — no path mints `¬(a R b)`.
- **`trusted/src/matching.rs` (1)** — `apply_rewrite`.

The transport calculus (refl/sym/trans/eq_mp/cong) is **ungated** — sound in
every language. The gated entries are `apply` and `canon` (checked against
`Language::admits` of the rule's own `TypeId`) and `execute` (checked against
`Rel<F>`'s `TypeId`). That gate — *"a rule can only fire in a language that
admits it"* — is what makes the per-language TCB be exactly its admitted set.

## The admitted rules, per language

An **admitted rule** is a trusted inference the base's `apply`/`canon` will fire
for a given `Language`. They are pinned in golden manifests (CI fails if the code
drifts from the file). Snapshot counts: **core 25, eval 17, builtins 387.**

**`core` (25) — the HOL kernel** (`docs/deps/core-manifest.txt`): HOL Light's
equality core + a few audited derived constructors —
`Refl Sym Trans MkComb Abs BetaConv EtaConv Assume EqMp DeductAntisym Inst
InstTFree Weaken` (the classic HOL Light primitives), plus the subtype/spec and
definitional rules `SpecAbsRep SpecRepAbsFwd SpecRepAbsBack SpecAx
UnfoldTermSpec Define NewTypeDefRule`, plus the two non-computational primitives
`UnitEq NatInduct` and the nat constructors `SuccInj ZeroNeSucc`, and `SelectAx`
(ε). Note `lem` and `false_elim` are **not** here — they left the kernel as
zero-TCB derived rules in `covalence-hol-eval`.

**`eval` (17) — the computation tier** (`docs/deps/eval-manifest.txt`): the
per-family *computation certificate* + *toHOL* rules —
`ToHolNatVal PairVal NatAddCert ToHolIntVal ToHolBytesVal ToHolF32Val
ToHolF64Val NatArithCert SuccCert IntArithCert BytesCert FixedWidthCert
LitEqCert CoercionCert FloatCert ZeroLitCert HolApp`. Each states an equation
whose right side is computed natively then certified — trust is per-rule via
`admits`, and the drivers around them (`reduce`/`prove_true`/`delta`) are
untrusted.

**`builtins` (387) — the native op catalogue** (`docs/deps/builtins-manifest.txt`,
`covalence-pure-eval`): every native computation the kernel will ever trust, as
per-op `Op`+`CanonRule` ZSTs (nat/int/bytes + the `u8`…`s64` fixed-width families
+ float conversions). Large because it enumerates the whole arithmetic surface;
each entry is a single, differentially-tested op semantics.

## Term/type shape (the vocabulary the TCB reasons over)

From `tcb-audit.json → globals`: **19 `TermKind` variants, 7 `TypeKind`
variants** — the fixed grammar of HOL terms/types the kernel manipulates.
Kept small on purpose: connectives (even `T`/`F`) are *defined constants*, not
term-kind variants (`defs/logic.rs`).

## The future: a parallel-structure, cross-referenced TCB doc

The maintainer's direction is a TCB doc that mirrors the *structure of the data*
and cross-references it, rather than restating it in prose. The pieces already
exist to make that live rather than snapshot:

- **`covalence-tcb-db`** (`crates/app/tcb-db`) already round-trips the golden
  manifests + audit into a STRICT-schema SQLite (`languages`, `language_rules`,
  `configs`, `mint_sites`) with semantic queries ("is `Refl` admitted by
  `CoreLang`?", "are all mint sites inside `trusted/src`?"). That `.db` is the
  natural backing for a queryable, always-current TCB doc.
- **The web TCB page** (`apps/covalence-web/src/routes/tcb/+page.svelte`) already
  renders `tcb-audit.json` (embedded at build, CI-gated by `bun run deps:check`)
  — the human view of the same data.
- **The relational base rewrite** ([`../design/relational-base-rewrite.md`](../design/relational-base-rewrite.md))
  turns axioms into serialized propositions, at which point a rule row becomes
  `(relation id, schematic proposition)` and the TCB doc can *link each admitted
  rule to its statement* — the parallel-structure, cross-referenced form.

The target: this prose doc becomes the **narrative index**, `tcb-db` the
**queryable body**, the web page the **live view**, and each admitted rule
cross-links from all three to its manifest line and (post-rewrite) its
proposition. Until then, this file is the accurate snapshot; the JSON is truth.
