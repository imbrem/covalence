- **Application shells (REPL, serve, client, LSP, VSCode, web)** bound

  [`MVP_DESIGN.md`](../../../../../MVP_DESIGN.md)).
- **Two competing planning docs.**
  [`prover-mvp-plan.md`](../../../../prover-mvp-plan.md) (rewrite
  `covalence-kernel` as new HOL kernel + add `covalence-shell`) vs the
  [`layered-framework`](../../layered-framework/) proposal (Framework
  / HOL / Morphism, ~800 LoC framework as TCB).
- **In-flight VCS work.** `covalence-fuse` scaffold (commits
  `b66c655` + `3f9207a`) — `cov cog mount` subcommand, FUSE
  scaffolding, readdirplus + file size realization.

User framing: *"very rough experimental code to try to sketch out the
high-level API with some preliminary implementations. Flesh out the
high level API with something behind it, then swap out the bottom and
build it up in a principled way. Bit by bit, until we converge to
something workable and practical. Breaking tests or even removing them
along the way is fine — but the end goal is to have something with a
very small TCB but lots of high-level API surfaces ingesting lots of
different proof formats."*

This framing — **wide-then-deep, with shells stubbed freely** — turned
out to be load-bearing for the strategy that emerged. See
[§3.5](#35-the-shell-strategy-as-architecture).

## 2. The diagnosis

The vision in
[`ARCHITECTURE.md`](../../../../../ARCHITECTURE.md) reads unified
("scoped truths about named artifacts, no global lies") but
operationally describes **three projects sharing infrastructure**:

1. **A theorem prover** — tiny LCF kernel + oracle discharge (§2–§4,
   §9 of ARCHITECTURE.md).
2. **A version-control / content-addressed substrate** — Git-trees +
   Dolt-style prolly tables + FUSE mount (§10–§11).
3. **A federated trust infrastructure** — mirrors, ElidePolicy,
   mount-as-implication, signed Thm exchange (§4, §10, §12).

The Git-repo theorem demo touches all three at once — rhetorically
beautiful, operationally too much for an MVP.

The unclarity was **not conceptual** (`ARCHITECTURE.md` is sharp).
It was that **more was in flight than fits in one head**:

- Three kernels actively or semi-actively maintained.
- Phase A–H refactor of the kernel that's also slated for rewrite.
- Phase P (EProp/EThm) adding to the kernel that's also being rewritten.
- Two competing rewrite plans.
- Application shells bound to a superseded API none of the new plans use.
- Multiple planning docs increasingly out of sync with each other.

The session was about closing this gap.

## 3. Reframings, in the order they emerged

The proposal went through several revisions mid-session as the user
corrected initial framings. Capturing them in order because the
*rejected* alternatives are useful context for future revisions.

### 3.1 Substrate-first with parallel streams (revised from prover-first)

**Initial recommendation.** Prover-first, with VCS deferred. Rationale:
the kernel is the smallest TCB, every other plane depends on it,
shipping it first gives you a real artifact.

**User correction.** *"The VCS and prover should be developed in
parallel, over the shared blob store and eventually tree store (since
we want to store our facts inside the tree store!), and eventually the
VCS will cease to be a separate thing, but will rather be a particular
theory inside the prover."*

**Why the correction is right.** (a) Content-addressing is first-class
for the prover (not bolted on later); (b) developing the VCS
separately exercises the blob store API which the prover needs anyway;
(c) the eventual unification ("VCS becomes a theory inside the
prover") needs both pieces to exist before it can land.

**What this changed in the proposal.** The path docs went from
"prover MVP + VCS later" to "substrate-first with two parallel streams
that share a content-addressed backbone." Phase 2 has both streams
running independently; Phase 3 stitches them at the substrate; Phase 4
is reunification.

### 3.2 `attest` / `decide` are not legacy

**Initial framing.** `attest()` / `decide()` from
[`MVP_DESIGN.md`](../../../../../MVP_DESIGN.md) treated as legacy code
that the rewrite deletes. Listed in
[`prover-mvp-plan.md`](../../../../prover-mvp-plan.md) "out of scope."

**User correction.** *"The attest/decide API is not completely legacy
either — we want to build this as a higher-level primitive, or use it
as an oracle primitive itself! So we need to think about that too —
the oracle primitive should probably be based on something like a
cleaned-up TreeStore crate."*

**What this means.** `attest`/`decide` are the **first concrete
instance of the general oracle primitive**. Specifically:

- The kernel's `attest()` host import is structurally
  `WASMExecutorAuthority.eval(component_hash, input_hash, output_hash)`
  — an oracle observation under an executor authority's `eval`
  relation.
- The kernel's `decide()` pipeline is structurally a userspace
  *strategy*: pick a decision procedure (a WASM component), drive it to
  attest, discharge the observation via a meaning-axiom, produce a Thm.
- The existing Wasmtime bridge, host-import plumbing, and component
  loading **survive the rewrite** under new ownership: in
  `covalence-oracle-wasm` rather than in the kernel.

**What this changed in the proposal.** Shared-backbone §4 captures
this; `MVP_DESIGN.md` got a banner marking it the WASM-oracle seed,
not legacy. The shells freeze; the attest mechanism relocates.

### 3.3 Verifiable reads and Stores leave the TCB (oracle-everything)

**Initial framing.** Verifiable reads as a TCB primitive (per
[`AGENTS.md`](../../../../../AGENTS.md)). Stores as framework
primitives (per
[`layered-framework/02-framework.md` §5](../../layered-framework/02-framework.md)
and
[`04-store.md`](../../layered-framework/04-store.md)).

**User correction.** *"Verifiable reads, and stores themselves, should
probably be oracles rather than being baked into the TCB. At least I
really like that idea — hence the proposed stratification — while it
makes things more complex at first it does mean we can freely extend
with other cryptographic schemes without swelling the TCB too much."*

**What this means.** A stronger LCF discipline: **only the framework
is in the TCB.** Hash functions, signature schemes, executors,
verifiable reads, store membership, merge certificates — *all*
oracles. The framework's only trust primitives are observations and
meaning-axioms.

**Trade-off acknowledged.** More upfront work to describe each
oracle's relations explicitly. Payoff: adding SHA-1, BLAKE3, IPFS,
Arweave, Git-LFS as new schemes is adding *oracle crates*, not TCB
code. The audit surface stays flat regardless of how many cryptographic
schemes accumulate.

**What this changed in the proposal.** Shared-backbone §3 captures the
stratification. The TCB collapses to *just the framework crate*;
Stores leave; verifiable reads become observations under store
authorities; merge certificates likewise.

### 3.4 Trusted store = untrusted store + hash oracle (the deeper version of 3.3)

**User added insight.** *"Compose untrusted stores as WASM components —
remember, a trusted store is just an untrusted store and a hash
oracle! Here the focus is on utility rather than correctness (it's
untrusted after all) — but eventually we want our kernel to carry an
untrusted blob store around + a hash function and use that to derive
all the other trusted store types (e.g. trees, etc.)."*

**What this means — strictly stronger than §3.3.**

- The minimal kernel substrate is **two things**: an untrusted
  blob-KV oracle, and a hash function oracle.
- A "trusted blob store" is a *derived theorem*: combine the
  blob-retrieval observation ("bytes B come from name N") with the
  hash observation (`hash(B) == N`) to conclude `verified_blob(N, B)`.
- Trees, prolly tables, format-tagged stores, signed-blob-stores all
  *derive further* from this. No `TreeStoreAuthority` is needed in
  the kernel; trusted-tree-API is a userspace derivation over
  blob+hash.
- Untrusted stores **compose as WASM components**: cache layer, retry
  layer, mirror layer, HTTP, IPFS, Git, SQLite — all WASM components
  implementing the same `BlobStore` interface, all freely composable,
  none in the TCB.
- Adding a hash scheme is adding **one oracle**, not a new store type.
  SHA-256 vs BLAKE3 vs Git-SHA-1 → same trusted-tree machinery on top,
  different hash oracle underneath.

**What this supersedes.** Shared-backbone §3's framing of
`BlobStoreAuthority` / `TreeStoreAuthority` as the trust-bearing
shapes. Under the deeper framing, those names refer to *interface
specs for untrusted stores*; the trust lives in the hash oracle and
the derived verification theorem.

**What this implies for the kernel's "carry around" state.** The
kernel needs to know about: one untrusted blob KV oracle (which it
calls to retrieve bytes by name), one hash function oracle (which
verifies retrievals). Everything else — trees, prolly, Parquet,
format tags, signatures — is constructed userspace.

**NOT YET written into the proposal.** Shared-backbone §3 still
reflects the §3.3 framing. Needs an update pass.

### 3.5 The shell strategy as architecture

**User stated.** *"covalence-shell implement[s] lots of untrusted high
level APIs (which I can always soundly stub / half-implement with
NotImplementedError) to play with, letting covalence-kernel evolve
freely."*

**What this means.**

- **`covalence-shell` is the untrusted-high-level-API home.** Its role
  is to expose ergonomic APIs the user wants to hack with, even when
  the kernel underneath isn't done.
- **Stub-with-NotImplementedError is a legitimate development mode.**
  Shells are outside the TCB; a stub that panics doesn't compromise
  soundness, it just means a particular operation isn't yet supported.
  As long as the kernel only constructs Thms via the rules, panicking
  shells are fine.
- **The kernel evolves freely behind the shell.** Shells absorb kernel
  changes; shell consumers (HOL API, OpenTheory, eventually SAT/SMT)
  don't have to.
- **Minimizes time-to-hack.** You can play with theorems via
  high-level APIs immediately, even if behind the curtain the kernel
  is half-built.

**Architectural pattern.** Classic ports-and-adapters: kernel ⇄
shell ⇄ format-specific shells (HOL, OpenTheory, SAT, SMT, Lean,
Metamath). Kernel changes ripple into the shell; the format-specific
shells stay stable.

**NOT YET written into the proposal.** Needs a dedicated short doc
(probably `01-shells-and-stubs.md`).

### 3.6 `covalence-hol` as the first shell (revised from "move to references/")

**Initial recommendation.** Move `covalence-hol` and
`covalence-opentheory` to `references/`; pin at known-good commit;
they're reference material for the eventual port.

**User correction.** *"I've got an agent right now porting
covalence-hol to be a shell over covalence-kernel, exposing the
standard HOL API. We want that because we want OpenTheory import — we
might even merge covalence-opentheory and covalence-hol into one
crate."*

**Why the correction is right.** (a) OpenTheory gives you tens of
thousands of mechanized theorems on day one — measurable demo target
and real corpus; (b) standard-HOL-API is the demand-shape; if the
kernel can support a HOL-Light-API shell, it can support the rest
(the shell *is* the requirements doc for the kernel, written in code);
(c) HOL is the **easiest** shell because the design target is
*modified HOL* — see [§7](#7-open-questions-for-the-interview) for
what "modified" means.

**Merge `covalence-opentheory` into `covalence-hol`: yes.** Single
direction of dependency; opentheory's only client is hol; stops the
artificial crate boundary.

**Caveat.** Route the HOL shell through `covalence-shell` (the
untrusted-API home from §3.5), not directly against `covalence-kernel`.
Then "kernel evolves freely" means "shell adapts when kernel changes;
HOL-API and OpenTheory don't have to."

**What this changed in the proposal.** The kill list for
`covalence-hol` / `covalence-opentheory` is **rescinded**. They stay
in the workspace under their new shell role. The merge is a future
move, not a Phase 0 action.

## 7. Open questions for the interview

Questions worth mining out of the user's head. Grouped by topic.

### Kernel design

- **What does "modified HOL" mean? What modifications?** *Partially
  answered* in the post-session followup; see
  [`../../layered-framework/notes/modified-hol.md`](../../layered-framework/notes/modified-hol.md).
  Summary: HOL + first-class observations + no privileged root context
  + disjunct-trick subset types; lives in the middle of a three-layer
  stack (foundation / standard-metatheory / current-theory); workflow
  is reverse-math-style morphism construction. Open follow-ups about
  E-graph / canonicalization modifications, whether observations are
  HOL-level or foundation-level, and which morphism toolkit is
  MVP-scope live in that doc's §7.
- **What kernel does the HOL shell port against right now?** The
  current `covalence-kernel` is mid-refactor with Phase A–H terminated;
  what's the actual API surface the shell is being written to?
- **What's the relationship between `covalence-kernel`'s legacy
  `Prop`/`Thm` and the framework's planned `Thm`?** The shell needs to
  expose one of them; which?
- **How does the kernel discover its blob-store and hash-function
  oracles at startup?** §3.4 says the kernel "carries" them; what's
  the wiring? Are they configured per-session, per-process, baked in
  at compile time?
- **Is the egraph (Phase P) approach permanently abandoned, or only
  inside the kernel?** Facts-not-proofs says "egraph as userspace
  oracle." Does that mean a future `covalence-oracle-egraph` crate?
  When does it land?

### Substrate and stores

- **Untrusted stores as WASM components — build now or after the
  kernel rewrite?** The WASM-component-composable-stores idea is
  exciting but a separate engineering project. Phase 1 (substrate
  hardening) or post-MVP?
- **What's the host responsibility when an untrusted store WASM
  component is malicious?** Returns wrong bytes? Returns no bytes?
  Infinite loops? Tries to escape sandbox? Does the host need
  defenses, or does soundness fall out of the hash-verification step?
- **What's the format-tagging story?** Bytes can be many things (tree
  nodes, prolly nodes, Parquet, WASM components, HOL theorems, signed
  blobs). When does format-plane stuff (`ARCHITECTURE.md` §12) become
  MVP-scope vs post-MVP?
- **Tree storage — Git-shaped only for V1, or prolly-trees right away?**
  Proposal Phase 2 currently leaves this open.

### Crate audit (status unknown)

For each, I don't currently know status:

- `covalence-forsp` — REPL syntax? Experiment? Cruft?
- `covalence-lean` — Lean ingestion shell? Stub? Cruft?
- `covalence-metamath` — Metamath ingestion shell? Stub? Cruft?
- `covalence-kv` — What is this? Mentioned nowhere I've seen.
- `covalence-sat`, `covalence-smt` — Active proof-format support, or

- `covalence-python` — PyO3 bindings, but bindings to *what*? The

### MVP scope

- **What's the acceptance test for the MVP?** Re-derive the OpenTheory
  standard prelude? A subset? "Run any OpenTheory article and produce
  Thm objects"? Pick a measurable threshold.
- **Is "import an OpenTheory article and re-export it as a signed Thm
  in the substrate" an MVP-or-post-MVP target?** It exercises the full
  loop (HOL shell → kernel → blob store oracle → format tagging) but

- **What's the "hackable" demo experience the user wants?** REPL? A
  programmatic Rust API? A Python notebook? FUSE-mounted theorem

### Project identity

- **One-sentence project tagline.** I proposed three alternatives

  substrate-first reframing but never confirmed the one-liner. Worth

- **What's the long-term home of `covalence-shell`?** A crate that
  stays the high-level surface forever, or a transitional crate that
  gets absorbed into per-format shells once they stabilize?

## 8. Philosophical commitments reaffirmed

These didn't change in this session but were re-stated as load-bearing:

- **Scoped truths about named, content-addressed artifacts; no global
  lies.** The north star from `ARCHITECTURE.md` §0.
- **Smallest possible TCB; everything checkable outside.** Pushed
  harder in this session than `AGENTS.md` currently codifies.
- **Three corners stay distinct.** Computation (executor) / logic
  (LCF) / evidence (oracle observations) per `ARCHITECTURE.md` §9.
  The shell strategy from §3.5 respects this: shells can be wrong, but
  they can't construct Thms outside the rules.
- **Facts, not proofs.** The bottom layer stores facts; soundness is
  constructional; algorithms above the line compose existing rule
  applications. Per
  [`facts-not-proofs.md`](../../layered-framework/notes/facts-not-proofs.md).
- **Mirror principle.** Confidence comes from agreement between
  independent paths. Not directly load-bearing for MVP but informs the
  long-tail federation work.

   proposal. Most of §3.4, §3.5, §3.6 needs incorporating.
2. Check §7 for open questions; if your revision touches one, capture
   the answer in the proposal and strike from the open-questions
   list.
3. Check §4 for the firm/tentative/directional labels; if you're
   reversing a firm decision, document why in a new session note.

When picking up the project after a break:

1. Read [`../00-overview.md`](../00-overview.md) first for the path.
2. Read these notes for the *trajectory* — what was tried, what got
   revised, what the user's intuitions were that didn't yet land in
   the proposal.
3. Skim §7 (open questions); these are the productive next
   conversations.
